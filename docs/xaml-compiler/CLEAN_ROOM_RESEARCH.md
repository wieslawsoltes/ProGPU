## Concepts rejected or deferred

- direct adaptation of another XAML compiler's AST, helper types, file structure, or emitter control flow: rejected by clean-room policy;
- runtime reflection as the normal type/member/binding engine: rejected for trim/AOT/performance reasons;
- treating successful XML parsing of Fluent resources as full compiler/runtime support: rejected; maturity is reported per phase;
- a single core assembly referencing all UI frameworks: rejected; profiles are separate packages.
- reflection-based resource factory discovery: rejected because it weakens trimming/AOT, type safety, deterministic registration, and startup behavior;
- one generated global registry source that directly references every resource factory: deferred because a single invalid artifact would prevent unrelated valid resource providers from registering; per-artifact typed registration is the current isolated publication model.
