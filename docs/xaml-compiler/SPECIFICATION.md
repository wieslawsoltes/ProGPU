# ProGPU XAML Compiler Official Specification

Version: `0.1-draft`
Status: normative implementation specification
Applies to: compiler core, Roslyn services, framework profiles, source generator, CLI, MSBuild/NuGet integration, hot reload, runtime object-model seams, tests, samples, and playground.

This document is the official entry point. The following Markdown documents are incorporated by reference and are normative parts of this specification:

1. [Normative requirements](REQUIREMENTS.md) — every `CR`, `SYN`, `INF`, `MKP`, `ROS`, `TYP`, `ATT`, `EDT`, `SEM`, `FWK`, `RES`, `FLU`, `GEN`, `HOT`, `BLD`, `CLI`, `TST`, `SMP`, and `DOC` requirement.

3. [Feature and maturity matrix](FEATURE_MATRIX.md) — the only valid statement of current support maturity.

No implementation behavior overrides these documents. If code and specification differ, the code is defective or the specification must be intentionally amended with tests and rationale.

## Governing decisions

- Roslyn is the foundational compiler object model. ProGPU reuses Roslyn text, span, location, diagnostic, annotation, symbol, compilation, workspace, project/document, change, C# syntax generation, simplification, and formatting capabilities.
- Generated C# exists as typed Roslyn syntax from construction through validation; string templates and parse-back generation shortcuts are prohibited. Text is produced only when serializing the final syntax tree for `AddSource`, files, or display.
- XAML syntax uses an isomorphic green/red model because Roslyn does not expose the `GreenNode` construction contract required for third-party `SyntaxNode` subclasses.
- Document XML and embedded markup-value languages have separate lossless Roslyn-style trees. The high-performance markup lexer/parser is a reusable `netstandard2.0` core service. Its immutable `XamlMarkupLanguage` registry validates versioned trigger/context/priority/associativity/conflict/capability declarations, integrates plugin tokenization, and requires custom root syntax to project into the canonical AST. Formatting and generation use registered inverses and accept their output only after parse/structural round-trip. Semantic and emission plugins remain independent later capabilities.
- The compiler is UI-framework-agnostic through the IR. WinUI, Avalonia, WPF, MAUI, and user dialects are versioned profile packages.
- Attribute metadata and supported CLR symbol shapes are first-class schema inputs. Profiles and users register deterministic mappings to neutral semantics; emitters never probe framework attributes or use untracked name heuristics.
- Build metadata is also schema: compilation mode, root namespace, source/resource identity, bindability, and complete-provider markers retain exact Roslyn evidence. Generator, CLI, workspace, and direct emission consume one immutable document decision with host-item, class, module, then assembly precedence.
- Symbol-shape registrations declare feature ownership and compose by feature rather than replacing whole framework policies. Incompatible equal-priority scalar/keyed declarations remain canonical diagnostics, while markup-extension callable/service rules and object-writer event-args rules stay scoped to the provider-qualified identity or annotation family that selected them.
- Text conversion is directional: type converters provide exact load-path create-from-string symbols, while value serializers remain independent save-path type/member metadata consumed by writers and bidirectional tooling through structured Roslyn syntax.
- Whitespace and initialization annotations are Roslyn-backed schema evidence. The infoset remains lossless; type-aware binding owns default normalization, inherited `xml:space`, significant-collection items, adjacent annotated-child trimming, and original/semantic text pairing. Explicit inherited Boolean overrides remain visible, and construction order changes only through a typed object-writer policy.
- Repeatable content-wrapper annotations are validated Roslyn symbol shapes. Directly assignable items bypass wrapping; foreign collection content requires an exact or unique most-specific typed wrapper and lowers through ordinary bound/IR construction rather than dynamic invocation.
- Constructor-argument annotations are exact save-path property-to-parameter mappings for serialization and inverse editing. They never alter ordinary load binding, and writer construction is generated only as structured Roslyn syntax from the selected constructor and parameter symbols.
- Designer and localization annotations are exact Roslyn schema evidence. Default values, serialization visibility/options, browsability/editor/design-time visibility, and localization category/readability/modifiability retain their original symbols and typed constants across source and metadata references. One neutral serialization policy determines excluded, element, content, or attribute form; these annotations never mutate ordinary load binding or construction IR.
- Convention-based designer serialization is an explicit provider-owned symbol shape. Exact `ShouldSerialize<Property>()` and `Reset<Property>()` candidates are validated and exposed as conditional/reset callables on the same neutral policy; compilation neither reflects over nor executes the target object.
- Every bound document exposes a deterministic stable-ID serialization-plan graph. It reuses canonical dependency ordering, assigns explicit omit/directive/attribute/element/content/constructor dispositions, retains effective serializer/default/conditional/reset symbols, and keeps constructor representation opt-in. Structured Roslyn factories and conservative reverse editing consume this graph; runtime value evaluation and final text writing remain separate services.
- Construction IR retains usable-during-initialization scheduling as neutral typed evidence. An ordinary marked child is declared, published through its canonical parent operation, and then populated; unmarked and non-created object kinds retain their established bottom-up schedule.
- Ordinary binding and compiled binding are different semantic families, each with an explicit schema role and distinct bound/IR values. An ordinary binding retains its constructed runtime descriptor, lossless shared path syntax, neutral source kind, and—when that source is provable—exact Roslyn source plus property/field or constant-indexer accessor evidence. Lexical `x:DataType`, explicit value, exact static member, exact resource-graph value, relative-self, same-namescope `ElementName`, and profile-resolved deferred-context sources remain distinct; any explicit runtime selector suppresses unrelated lexical evidence. Named source types are resolved in a post-bind immutable pass over the canonical namescope graph, giving forward and backward names equal treatment without leaking through deferred/template boundaries; unmatched names remain runtime-resolved. After the canonical resource graph is built, a neutral evidence service follows local definitions, aliases, same-typed variants, and external semantic-manifest candidates without constructing resources; only one common Roslyn type authorizes immutable resource-source enrichment, while the runtime lookup descriptor is preserved. Profiles classify deferred-context selectors and supply exact context types to both semantic binding and construction IR, so the core never recognizes WinUI `ControlTemplate`, `TargetType`, or `TemplatedParent`; WinUI uses that seam to type ordinary templated-parent paths. Integer `IList<T>` and string `IDictionary<string,T>` steps retain exact mutable indexer symbols and constant keys. Profiles publish accessors and immutable path plans only through structured statements and expressions; WinUI emits deterministic closed-generic static delegates in a generated module initializer plus `BindingPathSegment` construction, so generated paths require neither handwritten registration nor runtime parsing/reflection. Operation-valued markup extensions may request a framework-neutral registration owner through a typed profile contract; the core aggregates only explicit consumption evidence and orders structured preparation/finalization without knowing framework type names. WinUI ordinary bindings use that seam for both class-backed units and deferred factories, beginning before construction and attaching only after content and generated names commit so initial named/template lookup observes a complete tree. Compiled-binding vocabulary retains canonical Roslyn source/member/result/write/conversion/method/parameter symbols, and framework profiles publish it only as structured syntax. WinUI's compiled runtime consumes generated closed-generic static delegates, exact dependency-property identifiers, exact mutable collection-interface indexers, typed cast/function steps, retained attached accessors, and explicit owner/argument dependency paths—not path strings, reflection, or ordinary-binding accessor lookup. Both runtimes subscribe collection steps through typed notification contracts. Class-backed units expose the public profile-generated compiled `Bindings` lifecycle; construction defers activation until target/name/content commit, and hot reload replaces the source-owned group transactionally. Each deferred-factory invocation owns profile-generated ordinary and/or compiled controllers independent from its data context; the completed root initializes and owns every controller until direct, unloaded, or subtree release. Grammar capabilities not listed as implemented in the feature matrix remain explicit gaps.
- Framework create-from-string annotations and CLR type converters share one exact Roslyn callable descriptor with distinct static-factory versus converter-instance invocation kinds; text binding and structured emission never rediscover those callables by name.
- Resource keys are typed semantic values, never merely strings. Explicit keys, attribute-selected keys, profile-declared implicit keys, nested resource references, project manifests, structured Roslyn emission, and runtime lookup share one canonical identity model.
- Proven Roslyn constants and dictionary-converted literals compare by canonical CLR runtime value while retaining their original Roslyn expression identity for code generation; non-constant static values are never guessed equal.
- Cross-file resource imports retain lexical ownership. Provider publication, descendant visibility, nearest-scope shadowing, and incremental scope identity are explicit graph data rather than document-wide key heuristics.
- Conditional resources are sibling typed-key partitions, not flattened definitions. Profile-owned member roles identify resource vocabulary; manifests, provider slices, dynamic candidate sets, generated syntax, and incremental fingerprints preserve partition identity end to end.
- Compiled dictionary construction preserves the object-writer publication lifecycle. Types with exact usable-during-initialization evidence are attached before population; keyed resource-reference aliases remain emitted values; explicit provider reorder mode retains forward dependency edges without weakening the normal static-forward-reference diagnostic.
- Framework platform resources are a typed final lookup tier, not invented local definitions or nullable fallbacks. Alias IR retains its terminal key so profiles can select the correct Roslyn source type for structured conversion while runtime unknown keys still fail deterministically.
- Resource URIs preserve scheme, package/assembly authority, and path; authority-qualified identities are exact domains and are never reduced to global suffix matches.
- Editing and formatting are first-class compiler services, not playground-only utilities.
- Bidirectional editing is source-map and semantic-diff based. Reverse edits are applied only when an inverse is formally registered and unambiguous.
- Shipping packages are governed by one exhaustive project manifest. Portable packaging must pass an isolated local-feed consumer that imports transitive XAML build assets, loads only the source-generator assembly as an analyzer, compiles and runs generated bindings, and inspects persisted generated Roslyn output without repository project references or a shared NuGet cache.
- The production `ProGPU.WinUI.Themes.Fluent` package links and compiles the unchanged upstream theme. Current conformance covers syntax through complete dictionary load, partition-local resource roots, an extensible typed platform-resource seam, retained Light/HighContrast switching, and syntax-through-interaction execution of generated implicit Button and CheckBox templates. Untyped resource/setter/state/template-binding values share one invariant framework conversion boundary, including nullable numeric dependency properties. Supported state storyboards transactionally apply and restore typed final values while preserving dynamic resource markers. Full Fluent support still requires a native platform provider, non-dependency-object system-color mutation, duration/transition clocks, nested and non-visual target paths, specialized theme animations, broader control/accessibility behavior, hot reload, and visual tests.
- Clean-room implementation is mandatory.

## Conformance rule

A feature is conformant only when its requirements have automated evidence at the claimed maturity level. Tests must cite requirement IDs. Unsupported phases remain explicitly `Planned` or `Partial` in the feature matrix.
